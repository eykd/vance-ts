# ClawTask Product Requirements Document (PRD)

---

# Positioning Anchor

**ClawTask is the operating system that turns you and your AI into a disciplined OKR execution team.**

---

# 1. Executive Summary

ClawTask is an API-first, agent-native SaaS platform designed for power users who collaborate daily with AI systems (e.g., OpenClaw, Claude, GPT). It provides structural discipline, strategic alignment, and full auditability for commitments spanning work and life.

ClawTask is not a to-do list.
It is not a flexible workspace.
It is not a generic AI assistant.

It is a **Human–AI Commitment Operating System** that:

- Enforces structural execution discipline
- Links daily actions to quarterly outcomes
- Enables trusted agent mutation with full undo
- Reduces cognitive overload through behavioral nudges
- Provides executive-ready visibility into results

Primary outcome:

> The user walks into quarterly review calm, clear, and defensible.

---

# 2. Problem Statement

Modern managers embedded with AI face three compounding pressures:

1. Too many concurrent commitments ("too many plates spinning")
2. Manual translation between AI output and execution systems
3. Executive pressure for quarterly clarity (OKRs)

Current tools (Todoist, TickTick, Things, Notion, paper journals):

- Are human-only systems
- Do not support safe AI mutation
- Do not enforce structural next-action discipline
- Do not visibly link projects to quarterly outcomes
- Allow unlimited "active" commitments

This results in:

- Overload
- Review avoidance
- Commitment drift
- Strategic misalignment
- Anxiety during quarterly reporting

ClawTask solves this by becoming the structural execution substrate for human–AI collaboration.

# 4. Core Product Principles

1. **API-first by design**
   - Every UI capability exists at the application layer
   - API may exceed UI capability
   - UI and API are separate adapters over same domain layer

2. **Strict structural discipline**
   - One active next action per project
   - Required area and context
   - Explicit state transitions

3. **Shared human–AI authority**
   - Agents can mutate system fully
   - All mutations attributed
   - All mutations recoverable

4. **Transparency over restriction**
   - No sandbox mode
   - No hidden automation
   - Full audit trail

5. **Expose misalignment, do not coerce**
   - Nudges instead of hard limits
   - Alignment visibility over enforcement

6. **Single-user v1, expandable architecture**
   - Workspace abstraction from day one
   - No premature multi-user complexity

# 5. Functional Requirements

## 5.1 Core Entities

- InboxItem
- Action
- Project
- Area
- Context
- Outcome
- Cycle
- ReviewSession
- Comment
- Actor
- AuditEvent


## 5.2 Inbox & Clarification

- Separate InboxItem entity
- Clarification is atomic
- One InboxItem produces exactly one primary entity
- Clarified entity recorded via clarified_into_id


## 5.3 Projects & Actions

Rules:

- Flat project hierarchy
- One active next action per project
- Unlimited standalone active actions
- Projects must belong to Area
- Actions must have Context and Area
- Actions under project inherit Area
- No auto-completion of projects

Derived flags:

- needs_next_action
- ready_for_completion


## 5.4 Strategic Alignment Layer

Hierarchy:

Cycle → Outcome → Project → Action

Rules (v1):

- Projects may exist outside active Cycle
- Outcomes belong to one Cycle
- Projects may link to Outcomes via supports relation

Nudges:

- Active project not linked to Outcome
- Outcome with no supporting projects
- Active cycle ending soon

No hard enforcement.


## 5.5 Nudge Engine (Core Behavioral Layer)

Nudges are computed signals, not stored entities.

Categories:

Overload Nudges:
- Too many active projects
- Too many active standalone actions

Alignment Nudges:
- Active projects not linked to Outcome
- Outcomes without supporting projects

Discipline Nudges:
- Weekly review overdue
- Inbox not zero
- Projects missing next action

Strategic Nudges:
- Cycle nearing end date

Nudges must:

- Be queryable via API
- Be explainable
- Be dismissible (but recompute)
- Support UI surfacing


## 5.6 Review System

Types:
- Daily
- Weekly
- Monthly
- Quarterly

ReviewSession stores:
- type
- timestamps
- actor
- notes
- checklist_version

State computed dynamically.

Review not blocked by inbox count.


## 5.7 Agent Mutation & Undo

Agents have full mutation parity.

All mutations:

- Recorded in append-only audit log
- Attributed to Actor
- Timestamped
- Support compensating undo

Undo model:

- Entity-level undo
- State transition reversal via compensating command
- No destructive rewrite of history

# 6. Non-Functional Requirements

## 6.1 Architecture

- Clean Architecture / Hexagonal
- Domain layer pure
- Application layer enforces invariants
- UI and API as adapters
- Relational database
- Append-only audit log


## 6.2 Performance

- Sub-200ms API response for mutation commands
- Sub-150ms read queries under normal load


## 6.3 Reliability

- 99.5% uptime target (v1 SaaS)
- No data loss tolerance
- Audit log must be durable


## 6.4 Security

- API key-based authentication
- Scoped to workspace
- HTTPS-only
- Role abstraction present but single-user enforced in v1


## 6.5 Extensibility

- Workspace abstraction
- Actor abstraction
- Permission model extensible
- No UI-API coupling

# 7. MVP Scope (v1)

Include:

- Inbox + Clarify
- Projects + Actions
- Areas + Contexts
- One active next action enforcement
- Review sessions (weekly minimum)
- Nudge engine (overload + alignment + discipline)
- Single active Cycle
- Outcome linking
- Full audit log
- Undo capability
- API-first contract

Exclude:

- Multi-user collaboration
- SSO
- Enterprise compliance
- Advanced analytics dashboards
- Bulk operations

- % of active projects linked to outcomes

- Weekly review completion rate
- Average active projects per user
- Nudge resolution rate

Trust Metrics:

- % of mutations performed by agent
- Undo usage rate

Retention Metrics:

- 90-day retention
- Quarterly cycle completion rate

Qualitative Metric:

- User reports feeling calm entering quarterly review

# 9. Risks & Mitigations

Risk: System feels too rigid
Mitigation: Nudges over hard limits

Risk: Agent mutations cause fear
Mitigation: Full undo + visible attribution

Risk: Over-engineering before product-market fit
Mitigation: Single-user focus in v1

Risk: Users ignore nudges
Mitigation: High-signal surfacing in UI + CLI
