# ClawTask

## Comprehensive Product Specification (v2)

---

# Part I — Vision & Principles

## 1. Product Identity

ClawTask is an API-first, agent-centric GTD system designed for collaborative human–AI task management. It implements strict GTD semantics, enforces structural hygiene, and exposes strategic alignment through a 90-day execution layer.

ClawTask is not a to-do list. It is not a flexible workspace. It is a **Human–AI Commitment Operating System** — a disciplined execution substrate where humans and AI agents operate as peers over a shared commitment graph.

**Primary outcome:** The user walks into quarterly review calm, clear, and defensible.

## 2. Core Principles

1. **API-first by design.** Every capability exists at the application layer. The HTTP API and Web UI are separate adapters over the same domain. The API may exceed UI capability; the UI never exceeds API capability.

2. **Strict structural discipline.** The system enforces GTD semantics as domain invariants, not suggestions. One active next action per project. Required area and context. Explicit, named state transitions.

3. **Shared human–AI authority.** Agents can do anything the human can do. All mutations are attributed, timestamped, and reversible. No sandbox mode.

4. **Transparency over restriction.** Full audit trail. Visible attribution. No hidden automation. Nudges over hard limits.

5. **Expose misalignment, do not coerce.** The system makes strategic gaps visible through computed nudges with actionable affordances. It does not block the user from making conscious choices.

6. **Single-user v1, expandable architecture.** Workspace abstraction exists from day one. No premature multi-user complexity.

## 3. Architectural Commitment

ClawTask follows Clean Architecture (Hexagonal / Ports & Adapters) with Domain-Driven Design.

**Domain Layer (Pure):** Aggregates, entities, value objects, domain invariants, domain events. No framework dependencies. This is the heart of the system.

**Application Layer:** Command handlers (use cases), query services, nudge computation. Enforces GTD semantics and orchestrates domain operations. Both adapters depend on this layer.

**Adapter Layer:** HTTP API adapter (JSON), Web UI adapter (HTMX), Persistence adapter (relational DB). The Web UI calls the application layer directly — it does not call the HTTP API. Both adapters are peers consuming the same commands and queries.

```
┌──────────────────────────────────────────────┐
│              Adapter Layer                    │
│  ┌──────────┐  ┌──────────┐  ┌────────────┐  │
│  │ HTTP API │  │  Web UI  │  │ Persistence│  │
│  │ (JSON)   │  │  (HTMX)  │  │  (RDBMS)   │  │
│  └────┬─────┘  └────┬─────┘  └─────┬──────┘  │
│       │              │              │          │
│───────┴──────────────┴──────────────┴─────────│
│              Application Layer                │
│  ┌──────────────┐  ┌──────────────────────┐   │
│  │   Commands   │  │   Queries            │   │
│  │  (Use Cases) │  │  (incl. Nudge Engine)│   │
│  └──────┬───────┘  └──────────┬───────────┘   │
│─────────┴──────────────────────┴──────────────│
│              Domain Layer                     │
│  ┌────────────────────────────────────────┐   │
│  │  Aggregates · Invariants · Events      │   │
│  └────────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
```

**Persistence model:** Relational database with append-only audit log. Not full event sourcing — chosen for intelligibility.

**Deletion model:** Soft delete via `archived` status. Admin-only purge permanently removes entity and generates an audit event.

# Part II — Domain Model

## 4. Actors

Actors represent any entity performing mutations.

**Types:** `human`, `agent`

Agent credentials include a human-readable label (e.g., "Claude — daily triage").

All mutations record:

| Field | Description |
|-------|-------------|
| actor_id | Unique identifier for the actor |
| actor_type | `human` or `agent` |
| actor_label | Human-readable name |
| timestamp | When the mutation occurred |
| correlation_id | Optional grouping identifier |

Actors are first-class. There is no distinction in capability between human and agent actors — only attribution.

## 5. Workspace

Workspace is the top-level organizational boundary.

| Field | Description |
|-------|-------------|
| id | Unique identifier |
| name | Workspace name |
| created_at | Creation timestamp |

All entities belong to exactly one Workspace. Agents authenticate via API keys scoped to a Workspace. v1 supports a single user per Workspace.

## 6. Aggregates

ClawTask defines the following distinct aggregates. Each aggregate owns its own invariants and lifecycle. They are **not** subtypes of a shared Commitment base — they are independent aggregate roots (except where noted as owned entities).

### 6.1 InboxItem

The capture entry point. All new commitments begin here.

**Fields:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | UUID | Yes | Unique identifier |
| workspace_id | UUID | Yes | Owning workspace |
| title | string | Yes | Capture text |
| description | string | No | Additional detail |
| status | enum | Yes | `inbox` or `clarified` |
| clarified_into_type | string | Nullable | Type of entity produced |
| clarified_into_id | UUID | Nullable | ID of entity produced |
| created_at | timestamp | Yes | |
| updated_at | timestamp | Yes | |

**Invariants:**

- Status begins as `inbox`.
- Clarification is atomic: one InboxItem produces exactly one primary entity.
- Once clarified, the item is immutable (status = `clarified`, links recorded).
- Clarification cannot be undone by modifying the InboxItem — the produced entity must be archived independently.

### 6.2 Project (Aggregate Root — owns Actions)

A multi-step commitment. The Project aggregate is the **owner of its child Actions** and enforces the one-active-next-action invariant internally.

**Project fields:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | UUID | Yes | Unique identifier |
| workspace_id | UUID | Yes | Owning workspace |
| title | string | Yes | |
| description | string | No | |
| status | enum | Yes | See §7 |
| area_id | UUID | Yes | Must belong to exactly one Area |
| waiting_on | string | Nullable | Free text: what this project is blocked on |
| waiting_since | timestamp | Nullable | Auto-set on transition to `waiting` |
| scheduled_on | date | Nullable | Date-only scheduling |
| created_at | timestamp | Yes | |
| updated_at | timestamp | Yes | |

**Child entity — Action (owned by Project):**

See §6.3 for Action fields. When an Action belongs to a Project, it is an owned entity within the Project aggregate boundary. The Project loads its Actions to enforce invariants.

**Project invariants:**

- Must belong to exactly one Area.
- Flat structure — no project nesting.
- Exactly one child Action may have status `active` at any time. All others must be `ready`, `waiting`, `scheduled`, `someday`, `done`, or `archived`.
- Cannot transition to `done` unless all child Actions are `done` or `archived`.
- Project completion is always intentional — never automatic.
- Changing the next action is an atomic swap via the `SetNextAction` command on the Project aggregate.

**Derived flags (computed, not stored):**

- `needs_next_action` — Project is `active` but has no `active` child Action.
- `ready_for_completion` — Project is `active` and all child Actions are `done` or `archived`.

### 6.3 Action

A single concrete next step. Actions exist in two forms:

**Project Action:** Owned by a Project aggregate. Subject to the one-active-next-action constraint. Area is inherited from the parent Project.

**Standalone Action:** An independent aggregate root. Not subject to any next-action constraint. Unlimited standalone actions may be `active` simultaneously. Must specify its own Area.

| title | string | Yes | |
| description | string | No | |
| status | enum | Yes | See §7 |
| project_id | UUID | Nullable | If owned by a Project |
| area_id | UUID | Yes | Inherited from Project for project actions; required directly for standalone |
| context_id | UUID | Yes | Must have exactly one Context |
| waiting_on | string | Nullable | Free text: what this action is blocked on |
| waiting_since | timestamp | Nullable | Auto-set on transition to `waiting` |
| scheduled_on | date | Nullable | Date-only scheduling |
| created_at | timestamp | Yes | |
| updated_at | timestamp | Yes | |

**Invariants:**

- Must have exactly one Context.
- Must have exactly one Area (inherited from Project if applicable, explicit if standalone).
- A project Action's Area must match its parent Project's Area.
- Standalone actions have no project-level constraints on active count.

### 6.13 NudgeAcknowledgment

A persistent record of an actor's disposition toward a specific nudge condition.

**Fields:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | UUID | Yes | Unique identifier |
| workspace_id | UUID | Yes | Owning workspace |
| nudge_type | string | Yes | e.g., `projects_need_next_action` |
| entity_type | string | Nullable | Type of the entity triggering the nudge |
| entity_id | UUID | Nullable | ID of the entity triggering the nudge |
| disposition | enum | Yes | `resolved`, `archived`, `snoozed` |
| actor_id | UUID | Yes | Who acknowledged |
| snoozed_until | date | Nullable | Required if disposition is `snoozed` |
| created_at | timestamp | Yes | |

**Disposition semantics:**

- **Resolved:** "I've handled this." Nudge is suppressed. If the underlying condition recurs (was fixed and then broke again), the nudge resurfaces. The nudge engine checks whether the condition has been continuously true since the resolution timestamp — if not, it's a recurrence and the acknowledgment no longer applies.
- **Archived:** "I know, and I don't care." Nudge is permanently suppressed for this specific entity/type pair until the actor explicitly unarchives it. Auditable choice to override the system's opinion.
- **Snoozed:** "Not now." Nudge is suppressed until `snoozed_until` date. After that date, it resurfaces regardless.

All acknowledgment operations generate audit events.

### 6.14 AuditEvent

Append-only record of all state mutations.

**Fields:**

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| id | UUID | Yes | Unique identifier |
| workspace_id | UUID | Yes | Owning workspace |
| entity_type | string | Yes | Type of mutated entity |
| entity_id | UUID | Yes | ID of mutated entity |
| event_type | string | Yes | Named command that produced this event |
| payload_json | JSON | Yes | State change details (before/after) |
| actor_id | UUID | Yes | Who performed the mutation |
| correlation_id | UUID | Nullable | Optional grouping |
| timestamp | timestamp | Yes | |

**Invariants:**

- Append-only. No updates, no deletes.
- Every domain command produces at least one AuditEvent.
- payload_json contains sufficient state to compute compensating undo commands.

## 7. Status Model

### 7.1 Statuses

| Status | Applies To | Meaning |
|--------|-----------|---------|
| inbox | InboxItem only | Captured but unclarified |
| clarified | InboxItem only | Processed into a domain entity |
| ready | Action, Project | Clarified and available but not currently engaged |
| active | Action, Project, Area, Outcome, Cycle, Reference | Currently engaged |
| waiting | Action, Project | Blocked on external dependency |
| scheduled | Action, Project | Deferred to a specific date |
| someday | Incubation | Parked for future consideration |
| planned | Outcome, Cycle | Defined but not yet active |
| done | Action, Project | Completed |
| achieved | Outcome | Successfully delivered |
| abandoned | Outcome, ReviewSession | Intentionally dropped |
| completed | Cycle, ReviewSession | Finished |
| archived | All persistent entities | Soft-deleted |
| in_progress | ReviewSession | Currently being conducted |

### 7.2 Transition Rules

Status transitions are governed by **named commands** (see §8). There is no generic "change status" operation. Each transition has its own command with specific preconditions and side effects.

Key constraints:

- `archived` is reachable from any status (universal soft-delete).
- `done` requires all domain invariants to hold (e.g., Project cannot be `done` with non-terminal child Actions).
- Transition to `waiting` requires `waiting_on` to be provided and auto-sets `waiting_since`.
- Transition from `waiting` to any other status clears `waiting_on` and `waiting_since`.
- `scheduled` requires `scheduled_on` date.

# Part III — Command Surface

## 8. Commands

All state mutations flow through named commands in the application layer. Both the HTTP API adapter and the Web UI adapter invoke the same commands. Each command enforces its own preconditions, mutates domain state, and emits one or more AuditEvents.

### 8.1 Inbox Commands

| Command | Description | Preconditions |
|---------|-------------|---------------|
| CaptureInboxItem(title, description?) | Creates a new InboxItem with status `inbox` | — |
| ClarifyInboxItem(inbox_item_id, target_type, target_fields) | Atomically creates the target entity and marks InboxItem as `clarified` | Item must be in `inbox` status. Target type must be one of: action, project, incubation, reference |

### 8.2 Project Commands

| Command | Description | Preconditions |
|---------|-------------|---------------|
| CreateProject(title, area_id, ...) | Creates a new Project in `ready` status | Area must exist and be active |
| ActivateProject(project_id) | Transitions to `active` | Must be `ready` or `scheduled` |
| SetNextAction(project_id, action_id) | Atomically deactivates current active Action (if any) and activates the specified Action | Project must be `active`. Action must belong to this Project and be in `ready` status |
| SuspendProject(project_id, waiting_on) | Transitions to `waiting`, deactivates current next action | Must be `active`. `waiting_on` text required |
| ScheduleProject(project_id, scheduled_on) | Transitions to `scheduled` | Must be `ready` or `active`. Date required |
| ShelveProject(project_id) | Transitions to `someday` | Must not be `done` or `archived` |
| CompleteProject(project_id) | Transitions to `done` | Must be `active`. All child Actions must be `done` or `archived` (`ready_for_completion` must be true) |
| ResumeProject(project_id) | Transitions from `waiting` or `scheduled` back to `active` | Must be in `waiting` or `scheduled`. Clears waiting fields |
| ArchiveProject(project_id) | Soft-deletes | Any status |
| AddActionToProject(project_id, title, context_id, ...) | Creates a child Action in `ready` status | Project must exist. Area inherited from Project |

### 8.3 Action Commands (Standalone)

| Command | Description | Preconditions |
|---------|-------------|---------------|
| CreateStandaloneAction(title, area_id, context_id, ...) | Creates an independent Action in `ready` status | Area and Context must exist and be active |

|---------|-------------|---------------|

| SnoozeNudge(nudge_type, entity_type?, entity_id?, snoozed_until) | Creates acknowledgment with `snoozed` disposition | `snoozed_until` date required |

### 8.12 Administrative Commands

| Command | Description | Preconditions |
|---------|-------------|---------------|
| PurgeEntity(entity_type, entity_id) | Permanently removes an archived entity | Entity must be in `archived` status. Admin-only. Generates audit event |

### 8.13 Undo

| Command | Description | Preconditions |
|---------|-------------|---------------|
| UndoLastMutation(entity_type, entity_id) | Reverses the most recent undoable mutation on the specified entity by issuing a compensating command | Entity must have at least one undoable audit event. The compensating command must pass all current domain invariants |

Undo is **linear and per-entity.** The system derives the undo stack from the audit log for a given entity. Only the most recent undoable mutation can be reversed. The undo itself is a compensating command that produces its own AuditEvent — the audit log always grows forward, never rewrites.

If the compensating command would violate a current invariant, the undo fails with an explanation.

# Part IV — Query Surface

## 9. Queries

Queries are read-only operations in the application layer. They return computed state — no mutations. Both adapters consume the same query services.

### 9.1 Entity Queries

Standard CRUD reads for all aggregates, filtered by Workspace. Includes:

- Get entity by ID
- List entities by type (with status filters)
- List Actions by Project
- List Actions by Context
- List Actions by Area
- List Projects by Area
- List Outcomes by Cycle
- List Supports edges by Project or Outcome
- List Comments by entity
- List AuditEvents by entity

### 9.2 Scheduling Queries

Date-based views derived from `scheduled_on`:

- **Today:** Actions/Projects with `scheduled_on` = today
- **Tomorrow:** Actions/Projects with `scheduled_on` = tomorrow
- **Next 7 days:** Actions/Projects with `scheduled_on` within next 7 days
- **Overdue:** Actions/Projects with `scheduled_on` < today and status = `scheduled`

### 9.3 Derived Flag Queries

- **Projects needing next action:** Active Projects with no `active` child Action
- **Projects ready for completion:** Active Projects where all child Actions are `done` or `archived`
- **Incubation items due for review:** Incubation items with `review_after_date` ≤ today

### 9.4 Search

# Part V — Nudge Engine

## 10. Nudge System

The nudge engine is a **query-side computation** that evaluates current domain state, produces diagnostic signals, and attaches **actionable affordances** (suggested resolutions as domain commands) to each nudge.

### 10.1 Conceptual Model

A nudge consists of:

| Component | Description |
|-----------|-------------|
| nudge_type | Identifier for the condition (e.g., `projects_need_next_action`) |
| category | Classification (overload, alignment, discipline, strategic) |
| severity | `info`, `warning`, `critical` |
| message | Human-readable explanation |
| entity_type | The entity triggering the condition (if applicable) |
| entity_id | The specific entity (if applicable) |
| suggested_resolutions | One or more domain commands with pre-filled parameters that would resolve the condition |

**Nudges are not stored entities.** They are computed fresh on every query from current domain state, filtered against active NudgeAcknowledgments.

**Suggested resolutions are domain command references.** Each resolution describes a named command with its parameters — the same commands from §8. The HTMX adapter renders these as HATEOAS hypermedia controls. The JSON API adapter renders them as structured command payloads consumable by agents.

### 10.2 Nudge Categories and Thresholds (v1 — Hardcoded)

#### Overload Nudges

| Nudge Type | Condition | Severity | Threshold |
|------------|-----------|----------|-----------|
| too_many_active_projects | Count of active Projects exceeds threshold | warning | > 25 |
| too_many_active_standalone_actions | Count of active standalone Actions exceeds threshold | warning | > 15 |

**Suggested resolutions:** List the active Projects/Actions with affordances to `ShelveProject`, `CompleteProject`, `ArchiveProject`, `CompleteAction`, `ArchiveAction`.

#### Alignment Nudges

| Nudge Type | Condition | Severity |
|------------|-----------|----------|
| active_project_not_linked_to_outcome | An active Project has no Supports edge to any Outcome (only raised when an active Cycle exists) | info |
| outcome_without_supporting_projects | An active Outcome has no Supports edges | warning |

**Suggested resolutions:** For unlinked Projects — list active Outcomes with affordance to `LinkProjectToOutcome`. For unsupported Outcomes — list active Projects with affordance to `LinkProjectToOutcome`.

#### Discipline Nudges

| Nudge Type | Condition | Severity | Threshold |
|------------|-----------|----------|-----------|
| weekly_review_overdue | No completed weekly ReviewSession within threshold | critical | > 9 days |
| inbox_not_zero | Any InboxItem with status `inbox` exists | warning | > 0 items |
| projects_need_next_action | An active Project has no active child Action | critical | Per project |
| waiting_too_long | An entity has been in `waiting` status longer than threshold | warning | > 14 days |
| incubation_due_for_review | Incubation item's `review_after_date` ≤ today | info | Per item |

**Suggested resolutions:**
- `weekly_review_overdue` → `StartReview(weekly)`
- `inbox_not_zero` → `ClarifyInboxItem(inbox_item_id, ...)` for each item, with the InboxItem details
- `projects_need_next_action` → `SetNextAction(project_id, action_id)` listing the `ready` Actions available
- `waiting_too_long` → Entity details with `waiting_on` and `waiting_since` context for human/agent review

#### Strategic Nudges

| Nudge Type | Condition | Severity | Threshold |
|------------|-----------|----------|-----------|
| active_cycle_ending_soon | Active Cycle's end_date is within threshold | warning | < 14 days remaining |
| outcomes_not_scored | Cycle is being completed but Outcomes lack scores | warning | Per outcome |

**Suggested resolutions:**
- `active_cycle_ending_soon` → Links to active Outcomes for scoring
- `outcomes_not_scored` → `ScoreOutcome(outcome_id, ...)` for each unscored Outcome

### 10.3 Nudge Computation Pipeline

The nudge query executes the following pipeline:

1. **Evaluate all conditions** against current domain state, producing a raw nudge set.
2. **Load active NudgeAcknowledgments** for the Workspace.
3. **Filter:**
   - Remove nudges with `archived` acknowledgments (matching nudge_type + entity pair).
   - Remove nudges with `snoozed` acknowledgments where `snoozed_until` > today.
   - Remove nudges with `resolved` acknowledgments **unless** the condition has recurred (the condition was false at some point after resolution and is now true again).
4. **Attach suggested resolutions** to each surviving nudge by inspecting the relevant entities and pre-filling command parameters.
5. **Sort** by severity (critical → warning → info), then by category.
6. **Return** the computed nudge set.

### 10.4 Recurrence Detection (for Resolved Acknowledgments)

When a nudge has a `resolved` acknowledgment, the engine must determine whether the condition has **recurred** (was fixed and then broke again) or has been **continuously true** since resolution.

The mechanism: the engine checks the audit log for the relevant entity since the resolution timestamp. If any event indicates the condition was resolved (e.g., a next action was set, clearing `projects_need_next_action`) and then a subsequent event caused it to recur (e.g., the next action was completed with no replacement), the resolution is stale and the nudge resurfaces.

If no resolving event is found — meaning the condition has been continuously true since the acknowledgment — the resolved acknowledgment remains active and the nudge stays suppressed.

### 10.5 Adapter Rendering

**HTMX (Web UI):** Nudges render as HATEOAS hypermedia. Each suggested resolution is a link or form control pointing to the relevant command endpoint. The user clicks to act.

**JSON (HTTP API):** Nudges render as structured JSON objects with `suggested_resolutions` containing command name, parameters, and a human-readable description. Agents parse these as actionable directives.

Example JSON rendering:

```json
{
  "nudges": [
    {
      "nudge_type": "projects_need_next_action",
      "category": "discipline",
      "severity": "critical",
      "message": "Project 'Q3 Platform Migration' has no active next action.",
      "entity_type": "project",
      "entity_id": "proj_abc123",
      "suggested_resolutions": [
        {
          "command": "SetNextAction",
          "params": {
            "project_id": "proj_abc123",
            "action_id": "act_def456"
          },
          "description": "Activate 'Draft migration plan' as next action"
        },
        {
          "command": "SetNextAction",
          "params": {
            "project_id": "proj_abc123",
            "action_id": "act_ghi789"
          },
          "description": "Activate 'Review dependency matrix' as next action"
        }
      ]
    }
  ]
}
```

### 10.6 Review Session Integration

When a ReviewSession of type `weekly` is completed:

- All `resolved` NudgeAcknowledgments for `weekly_review_overdue` are refreshed (the completion event itself resolves the condition).
- The timestamp of the completed review becomes the new baseline for the `weekly_review_overdue` computation.

This ensures the review cadence nudge resets cleanly after each completed review.

# Part VI — Undo System

## 11. Undo Model

### 11.1 Philosophy

Undo in ClawTask is **linear and per-entity.** The audit log provides a complete, ordered history of mutations for every entity. Undo walks backward through this history one step at a time.

Undo never rewrites history. It issues a **compensating command** — a new forward-moving mutation that reverses the effect of the previous one. The audit log records both the original command and the undo.

### 11.2 Mechanics

1. Given an entity (type + id), retrieve its audit event stack in reverse chronological order.
2. Identify the most recent undoable event.
3. Compute the compensating command from the event's `payload_json` (which captures before/after state).
4. Execute the compensating command through normal domain validation.
5. If the compensating command passes all invariants, it succeeds and is recorded as a new audit event of type `undo_{original_event_type}`.
6. If the compensating command would violate a current invariant, the undo fails and returns an explanation.

### 11.3 Undo Availability

Each entity exposes a derived flag: `can_undo` (boolean) and `undo_description` (human-readable summary of what would be reversed). This information is available in the entity query response and can be rendered as an affordance in both adapters.

### 11.4 Non-Undoable Operations

Some operations are inherently non-undoable:

- PurgeEntity (permanent deletion — no state to restore)
- AuditEvent creation (append-only by definition)

All other commands are undoable in principle, subject to invariant validation at undo time.

# Part VII — Scheduling & Incubation

## 12. Scheduling

Scheduling uses **date-only** granularity (`scheduled_on` field). ClawTask is not a calendar. The calendar remains sacred per GTD principles.

When an Action or Project is scheduled:

- `scheduled_on` date is set.
- Status transitions to `scheduled`.
- The entity appears in date-based scheduling queries (Today, Tomorrow, Next 7 days, Overdue).

When the scheduled date arrives, the entity does not auto-transition. It remains in `scheduled` status and appears in the "Today" view. The actor (human or agent) must explicitly transition it to `active` or reschedule it. This is intentional — ClawTask does not perform implicit state transitions.

Overdue items (`scheduled_on` < today, status still `scheduled`) are surfaced in the Overdue query.

## 13. Incubation

Incubation items (Someday/Maybe) are parked ideas with a `review_after_date`. When the date is reached, the item surfaces in the incubation review query and triggers the `incubation_due_for_review` nudge.

The actor can then clarify the incubation into a Project or Action (via a new ClarifyInboxItem-style command), defer it with a new `review_after_date`, or archive it.

# Part VIII — Search, Comments, and Cross-Cutting Concerns

## 14. Search

Simple full-text search across `title` and `description` fields of all entity types within the Workspace.

Single query endpoint. No advanced filtering, faceting, or ranking in v1. Results return entity type, id, title, and a text snippet.

## 15. Comments

Comments provide narrative context on any entity. They are lightweight, markdown-ready, and editable.

Comments are **not** aggregate roots — they are standalone entities associated with a target entity via `entity_type` + `entity_id`. They do not affect the target entity's state or invariants.

Every comment edit produces an audit event capturing the before/after body.

## 16. Audit Trail

The AuditEvent log is the system's memory. It records every mutation performed by every actor. It is append-only and durable.

The audit log serves three purposes:

1. **Transparency.** Any actor can inspect the full history of any entity.
2. **Undo.** Compensating commands are derived from audit event payloads.
3. **Accountability.** Every mutation is attributed to a specific actor.

The audit log does not serve as an event store for projections — the system uses direct state reads against the relational database. The audit log is a historical record, not a source of truth for current state.
