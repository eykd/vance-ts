**Feature**: 012-clawtask-vertical-slice
**Branch**: `012-clawtask-vertical-slice`
**Date**: 2026-03-02

## Decision 1: better-auth Workspace Provisioning Hook

**Decision**: Use `databaseHooks.user.create.after` in `getAuth(env)` within `src/infrastructure/auth.ts`.

**Rationale**:

- The existing codebase already uses `databaseHooks` for `session.create.before` and `verification.create.before` — this is the established pattern for side effects inside better-auth's DB layer.
- The `after` variant fires after the user row is committed to D1, so `user.id` is stable and can be used as a FK.
- The provisioner has access to `env.DB` via the closure in `getAuth(env)`, which already captures this binding — no new wiring needed.
- Running synchronously within the signup request satisfies the spec assumption that "no user can authenticate without a workspace."

**Fallback (if `databaseHooks.user.create.after` is not available in the installed better-auth version)**:
Use a custom plugin with a path-matched `after` hook:

```typescript
plugins: [
  {
    id: 'workspace-provisioner',
    hooks: {
      after: [{ matcher: (ctx) => ctx.path === '/sign-up/email', handler: provisionHandler }],
    },
  },
];
```

Verify against `node_modules/better-auth/dist` during implementation.

**Alternatives considered**:

- Post-signup API call from the client: rejected — not atomic, leaves window where user has no workspace.
- Background job via Cloudflare Queue: rejected — async, violates "no user can authenticate without a workspace" invariant.
- Trigger in `SignUpUseCase`: rejected — application layer should not know about workspace provisioning infra.

## Decision 2: Atomic Transaction for ClarifyInboxItemToAction

**Decision**: Use `env.DB.batch([stmt1, stmt2, ...])` — the native D1 batch API — for the ClarifyInboxItemToAction use case.

**Rationale**:

- The spec explicitly states: "Atomic transactions for `ClarifyInboxItemToAction` are implemented using D1 batch operations."
- D1 batch is atomic: all statements succeed or none are applied.
- Drizzle's `db.transaction()` on D1 may not be available or reliable under the `wrapD1ForDrizzle` proxy used in the project.
- Raw `env.DB.batch()` with `db.prepare(...).bind(...)` is the D1-native pattern for atomicity.

**Implementation pattern**:

```typescript
const results = await env.DB.batch([
  env.DB.prepare('UPDATE inbox_item SET status=?, updated_at=? WHERE id=? AND workspace_id=?')
    .bind('clarified', now, inboxItemId, workspaceId),
  env.DB.prepare('INSERT INTO action (id, workspace_id, ...) VALUES (?, ?, ...)')
    .bind(actionId, workspaceId, ...),
  env.DB.prepare('INSERT INTO audit_event (...) VALUES (...)')
    .bind(...),
  env.DB.prepare('INSERT INTO audit_event (...) VALUES (...)')
    .bind(...),
]);
```

**Alternatives considered**:

- Drizzle `db.transaction()`: rejected — not confirmed to work correctly under `wrapD1ForDrizzle` proxy; D1 HTTP API historically lacked full transaction support.
- Two sequential D1 writes: rejected — not atomic; partial failure leaves data inconsistent.

## Decision 3: Drizzle Schema for New Tables

**Decision**: Define all new tables in `src/infrastructure/schema/clawtaskSchema.ts` using `sqliteTable()` from `drizzle-orm/sqlite-core`. Mirror the pattern in `src/infrastructure/authSchema.ts`.

**Conventions**:

- Column names: `snake_case` in D1, camelCase in Drizzle TypeScript projections.
- IDs: `TEXT` (`string` mode) — UUID generated by `crypto.randomUUID()` server-side.
- Timestamps: `TEXT` — ISO-8601 strings (same as auth schema; D1 does not have a native `DATETIME` type with timezone).
- Nullable references: typed as `TEXT` with `.nullable()`.
- Enums: stored as `TEXT` CHECK constraints in SQL; enforced as TypeScript string literals in domain.

**Alternatives considered**:

- Integer IDs (auto-increment): rejected — spec requires UUIDs; UUIDs are more secure and portable.
- Separate schema files per table: rejected — one schema file per feature slice is simpler and aligned with the existing `authSchema.ts` pattern.
- Drizzle enum type: rejected — Drizzle SQLite enum uses CHECK constraints anyway; explicit TS union types are more readable.

## Decision 4: Workspace Resolution Middleware

**Decision**: Add a `requireWorkspace` middleware (after `requireAuth`) that resolves workspace and actor from the authenticated user. Both are set in Hono context variables for downstream handlers.

**Rationale**:

- Every `/api/v1/*` and `/app/*` handler needs `workspaceId` and `actorId` for scoping and audit events.
- Resolving once in middleware avoids N+1 queries across handlers.
- Consistent with the existing `requireAuth` pattern (sets `user` and `session` in context).

**Context variables added**:

```typescript
Variables: {
  user: AuthUser;
  session: AuthSession;
  workspace: {
    id: string;
  } // minimal; expands if needed
  actor: {
    id: string;
  } // minimal; expands if needed
}
```

**Error handling**:

- If workspace not found for an authenticated user: return 503 (workspace provisioning may have failed) — matches the edge case in the spec ("user created but no workspace → 503 or error page").

**Alternatives considered**:

- Look up workspace in each handler: rejected — repetitive, error-prone, harder to test.
- Cache workspace in session token: rejected — over-engineering; D1 reads are fast.

## Decision 5: HTMX UI Pattern

**Decision**: Full pages at `/app`, `/app/inbox`, `/app/actions`. HTMX partials at `/app/_/inbox`, `/app/_/actions`, `/app/_/inbox/:id/clarify`, etc. All HTML templates are pure TypeScript functions returning `string`.

**Rationale**:

- Consistent with existing `AppPartialHandlers` pattern and the `/app/_/*` convention in `worker.ts`.
- No client-side JS state management needed; HTMX handles all partial updates.
- DaisyUI 5 components for forms, badges, and loading states (already in the project via Hugo static assets).

**HTMX patterns**:

- Quick capture form: `hx-post="/app/_/inbox" hx-target="#inbox-count" hx-swap="outerHTML"` — updates count in dashboard header.
- Clarify inline form: `hx-get="/app/_/inbox/:id/clarify-form"` expands the row; `hx-post="/app/_/inbox/:id/clarify"` swaps the row out.
- Activate button: `hx-post="/app/_/actions/:id/activate" hx-target="closest tr" hx-swap="outerHTML"` — replaces the row.
- Complete button: `hx-post="/app/_/actions/:id/complete" hx-target="closest tr" hx-swap="outerHTML"`.

**Alternatives considered**:

- Alpine.js client-side state: rejected — HTMX alone is sufficient for this feature; no complex client state needed.
- SPA-style routing: rejected — HTMX server-driven approach is the project's established pattern.

## Decision 6: Error Envelope Format

**Decision**: All API errors return `{ "error": { "code": "...", "message": "..." } }` as required by FR-014. HTTP status: 400 (validation), 404 (not found), 422 (business rule violation), 500 (unexpected).

**Error codes**:
| Code | HTTP | Meaning |
|------|------|---------|
| `validation_error` | 400 | Missing or malformed input |
| `not_found` | 404 | Entity not found in workspace |
| `invalid_status_transition` | 422 | State machine violation |
| `area_not_active` | 422 | Area is archived |
| `area_not_found` | 422 | Area ID does not exist in workspace |
| `context_not_found` | 422 | Context ID does not exist in workspace |
| `service_error` | 500 | Unexpected infrastructure failure |

## Decision 7: Workspace Seed Data

**Decision**: Seed 3 Areas and 5 Contexts during workspace provisioning. All names are constants in the domain.

**Default Areas**: `Work`, `Personal`, `Admin` (all status=`active`)
**Default Contexts**: `computer`, `calls`, `home`, `errands`, `office`

**IDs**: Generated at runtime with `crypto.randomUUID()` for each new workspace — not hardcoded.

**Alternatives considered**:

- Hardcoded UUIDs for seeds: rejected — UUIDs would repeat across workspaces, breaking FK uniqueness.
- Lazy seeding (create on first use): rejected — spec requires seeds to exist immediately after signup (US01 SC-001).
